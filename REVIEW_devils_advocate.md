# Devil's Advocate Review: Maynard M_{105} > 4 Audit Project

## 1. "Every computational fact is machine-verified" -- FALSE

The claim in STATUS.md's table -- that the "42-step PRS chain recurrence"
is machine-verified -- is misleading. What CRTCheck.v actually proves is
that the PRS identity holds **modulo 10 specific primes**. This is a
necessary condition for the identity to hold over Z, not a sufficient one.

More critically: `shipped_chain_eq` (the lemma asserting that the shipped
chain data in WitnessChain.v equals the chain that BrownTraub.sturm_chain
would compute from charpoly_int) is **Admitted** in CertL1.v (line 120).
Without it, the entire sign-verification pipeline (CRTSigns.v) only proves
that the *shipped* data is internally consistent -- it says nothing about
whether that data matches what the Rocq-side BrownTraub algorithm would
produce. The signs could be consistent with fabricated chain data, and the
proof would still go through.

This gap is partially disclosed in CertL1.v's header comments but is
**not disclosed** in the STATUS.md summary table, which lists sign
verification as fully "Qed, no admits" without caveat.

## 2. "CRT over 9,776 Uint63 primes" -- PHANTOM CLAIM

STATUS.md line 32 states: "42-step PRS chain recurrence | CRT over 9 776
Uint63 primes | 21 s | CRTCheck.v". STATUS.md line 132 lists
"CRTPrimes.v | 211 KB | 9 776 primes for CRT verification" as an
autogenerated witness file.

**CRTPrimes.v does not exist.** It is not on disk, not in `_CoqProject`,
and not in any `_CoqProject`-listed file. The actual CRTCheck.v defines
exactly **10 primes** inline (lines 195-206), giving ~300 bits of CRT
coverage. The maximum coefficient in the chain is ~293,000 bits. The 10
primes prove that the PRS identity holds modulo ~300 bits of information.
This is a probabilistic sanity check with a false-positive probability of
roughly 2^{-300}, not a mathematical proof.

CRTCheck.v's own comments (lines 182-189) honestly disclose this: "10
primes give ~300-bit CRT coverage -- a strong probabilistic check but not
a full CRT proof." But STATUS.md's table contradicts this by claiming
9,776 primes. This is either an aspirational claim that was never
implemented, or a documentation error. Either way, it is an overstatement.

Furthermore, even the CRT-to-Z lifting argument (`crt_correctness`, line
262) proves `True` -- it is a vacuous placeholder, not a proof.

## 3. "Bridge.v has 0 admits" -- TRUE BUT MISLEADING

Bridge.v indeed contains zero `Admitted` lemmas as of the current code.
However, the obligation that Bridge.v previously carried --
`mods_int_morph`, the strict chain equality between the concrete
Brown-Traub chain and MathComp's abstract `mods` -- was **moved** to
CertL1.v as `chain_is_mods` (line 272, Admitted). Bridge.v's header
comments (line 20-22) acknowledge this: "mods_int_morph removed: strict
chain equality is unprovable; consumers rewired to take Habs_chain as a
hypothesis instead."

The honest statement is: "Bridge.v has 0 admits because it parameterizes
over the hard obligation, which is now Admitted in CertL1.v." Saying
"Bridge.v has 0 admits" without this context gives the false impression
that the L1 Sturm bridge is complete. It is not.

## 4. "Headline has 2 axioms" -- TRUE, WITH CAVEATS

`Print Assumptions maynard_eigenvalue_S1` would indeed show
`sturm_count_correct` and `charpoly_int_eq_charpoly`. STATUS.md (line 75)
parenthetically notes "plus standard Uint63 kernel primitives." This is
honest but easy to miss. The Uint63 axioms (land, lor, lxor, etc.) are
trusted kernel primitives that underpin all vm_compute on native integers.
They are standard and accepted by the Rocq community, but a truly
skeptical reader should know the TCB includes these.

Additionally, `A_rat_unitmx` (Cert.v line 49) is Admitted but is NOT in
the transitive closure of the headline theorem -- it is used nowhere in
the proof of `maynard_eigenvalue_S1`. This is correct but worth noting:
the headline theorem does not actually prove that A_rat is well-defined as
a matrix inverse; it only proves that a certain polynomial (defined from
shipped data) has a root above 4/105 that is an eigenvalue of whatever
`invmx` returns (which is total, returning 0 for singular matrices).

## 5. The FLINT pipeline is unverified

`build_certificate.py` is a ~300-line Python script that calls
python-flint. It is not formally verified. The README discloses this
implicitly ("This layer runs the audit and produces the witness data")
but does not state plainly: "the FLINT layer is trusted; its output is
only checked to the extent that the Rocq layer can re-verify the shipped
data." Since `shipped_chain_eq` is Admitted, the Rocq layer currently
does NOT re-verify the shipped chain data against its own computation.
The only re-verification is the 10-prime modular check, which is
probabilistic.

## 6. "Independent re-implementation" -- PARTIALLY TRUE

The FLINT pipeline re-implements the Mathematica notebook's matrix
construction from closed-form Beta integrals. But the spot-checking is
thin: 4 entries of M1 and 1 entry of M2 are checked against closed-form
formulas (build_certificate.py lines 125-139). The 42x42 matrices have
1,764 entries each. A systematic error in the `prime_gram` integral
assembly (the hot path that builds all entries) would not be caught by
5 spot checks if the error only manifests at higher basis indices. The
symmetry check catches transposition bugs but not formula errors.

The claim "independently auditable" is fair for the pipeline architecture.
The claim "independent re-implementation" is fair for the code. But
"independently verified" would be an overstatement -- the verification
of the matrix entries themselves relies on trusting FLINT's rational
arithmetic and the Python translation of Maynard's formulas.

## 7. What this project ACTUALLY delivers

Strip away the framing. Here is what a skeptical mathematician gets:

This project provides a Rocq theorem whose statement (exists an eigenvalue
of a specific 42x42 rational matrix above 4/105) is precisely the
computational content of Maynard's Proposition 4.3. The theorem compiles
with Qed but depends on two Admitted axioms: (L1) that the shipped
characteristic polynomial has a real root above 4/105, and (L2) that the
shipped polynomial equals the actual characteristic polynomial of the
matrix. Neither axiom is discharged. Beneath them lie three further
Admitted lemmas in CertL1.v (`shipped_chain_eq`, `chain_is_mods`,
`no_root_at_cb`) and significant outstanding work on L2. The sign data
is verified against shipped (not computed) chain data. The PRS chain is
checked modulo 10 primes (~300 bits), not the ~293,000 bits needed for
a complete CRT proof, despite STATUS.md's claim of 9,776 primes. The
project is an impressive scaffold and architecture for a future complete
proof, but it is not, today, a machine-checked proof of M_{105} > 4.
